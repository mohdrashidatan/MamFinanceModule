## What Was Decided

### Academic Levels
12 levels: Primary 1–6, Secondary 1–4, Pre-University 1–2.

### Fee Types
| Code | Name | Frequency |
|------|------|-----------|
| F | Tuition Fee | Monthly |
| C | CCA Fee | Semi-Annual (March + July) |
| E | Enhancement Fee | Quarterly, PreU only |
| M | Miscellaneous Fee | Annual, $120 all levels |

### Invoice Number Format
`YYMM[Type]-[StudentID]` — e.g. `2601F-292393` = January 2026 Tuition for student 292393

### Architecture Chosen: Approach C — Structured Rule Tables + Service Layer
Fee rules stored in proper relational DB tables. A `feeRuleService` on the backend computes eligibility and amounts. No hardcoded rules, no JSON blobs.

### Key Design Decisions
- **One invoice per student per fee type per billing period** (not consolidated)
- **Duplicate prevention** at DB level: composite UNIQUE on `(student_id, fee_type_id, billing_calendar_id)`
- **Balance due** computed on read (`amount_after_reduction − amount_paid`), never stored
- **Payments** support: multi-invoice selection, partial payment, advance payment top-up, credit wallet apply
- **Reductions**: grants, scholarships, bursaries via `invoice_adjustments` table. Floor at $0 (can't go negative)
- **PDF generation** via pdfkit (server-side, no headless browser)

## Database: 15 Tables

### New Lookup Tables (6)
1. `academic_years` — active academic year control
2. `levels` — PRI1–PRU2 with programme and sort order
3. `fee_types` — F, C, E, M with frequency
4. `fee_rates` — amount per level per period per year (one row per level)
5. `billing_calendar` — what gets billed when + `is_processed` lock flag
6. `billing_calendar_levels` — which levels are billed for each calendar entry
7. `counter_items` — Zakat, Fitrah, Donation, Item Purchases (configurable)

### New Invoice Tables (4)
8. `invoice_batches` — groups invoices from one generation run
9. `invoices` — one per student per fee type per billing period
10. `invoice_adjustments` — grants/scholarships applied to an invoice
11. `receipt_items` — line items on a receipt (invoice payments + counter items)

### New Credit Table (1)
12. `student_credit_ledger` — ledger of advance payments, overpayments, credits applied

### Extended Existing Tables (2)
13. `students` — add `current_level_id`, `academic_year_id`
14. `receipts` — add `total_amount`, `credit_applied`, `payment_method`, `paid_at`, `created_by`
